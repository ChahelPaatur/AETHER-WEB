Features

Everything it does, out of the box.

Drive motors, read sensors, move servos — all from English. No bespoke code per task. Here's the full surface area.

F.01

Plain-English control.

The LLM planner translates objectives like "drive forward and stop 10 cm before the wall" into coordinated tool calls.

F.02

Auto-discovery.

Scans I²C, GPIO, serial, camera, audio, and network at boot, matches detected hardware to known adapters, and registers the right tool vocabulary automatically.

F.03

True parallel skills.

A single objective can spawn coordinated threads. drive_until_obstacle runs a motor thread and an ultrasonic-polling thread concurrently with a shared safety event.

F.04

Five-layer safety chain.

Ctrl-C, SIGTERM, crash, segfault — every path leads to motors stopped, pins released. There is no path that leaves the robot running.

F.05

Self-correcting.

The CorrectionAgent catches LLM hallucinations, reroutes via a fallback chain, and verifies outcomes — skipping verification on clean success to keep latency low.

F.06

Drone + ground robots.

MAVLink flight controllers auto-discovered on serial. Differential-drive cars, Mecanum platforms, pan-tilt rigs — all through one tool vocabulary.

F.07

Lightweight.

Runs on a Raspberry Pi 4. No GPU required. Inference happens via the Anthropic API; the Pi only orchestrates.

F.08

Universal adapters.

If your robot has a Python library, AETHER can drive it — via built-in adapters, offline-generated adapters, or a guided REPL.

The moat

Three-tier universality.

1
Built-in adapters for known kits

SunFounder PiCar-X, Freenove, Adeept, PCA9685, L298N, Waveshare — supported out of the box, zero configuration.

2
Generated adapters for new kits

Claude reads your kit's vendor Python library, generates a new adapter offline, validates it through a static safety check, and registers it. No vendor integration needed.

3
Guided REPL for custom hardware

Fully custom rigs are wired up interactively through a guided REPL that maps your hardware onto the standard vocabulary.

⌁

If your robot has a Python library, AETHER can drive it. Tier 2 generated adapters that try to register tools outside the standard vocabulary fail static validation and are rejected.

Architecture

How it works.

You type an English objective. The LLM planner picks tools from a registered vocabulary. Tools come from three sources: built-in primitives (cross-platform), kit adapters (Tier 1 or Tier 2 generated), and parallel skills (multi-thread coordinated capabilities).

Every tool call goes through the tool registry, executes against the hardware, and returns a result. The CorrectionAgent inspects each result — clean success advances to the next step; failure triggers a fallback chain. When the objective is done, the safety chain ensures motors are stopped and pins are released.

Design rationale

Why it's built this way.

Why parallel threads, not a serial loop.

A serial loop ("motor for 100 ms, read sensor, motor for 100 ms…") causes physical jitter — the motor accelerates and decelerates 10 times a second. True parallel threads run the motor continuously while a separate thread polls the sensor. Smooth motion, accurate stops, faster response.

Why a five-layer safety chain.

Robotics failures are unforgiving. A crashed process that left motors running could damage the robot, the environment, or a person. Each layer catches a different failure class:

FailureCaught by
Tool exceptionLayer 1 — tool try/finally
Skill mid-thread exceptionLayer 2 — skill try/finally
Ctrl-C in the main threadLayer 3 — _GLOBAL_STOP event
os._exit() or segfaultLayer 4 — atexit
Next process can't claim pinsLayer 5 — GPIO cleanup
  1. Every motion tool wraps in try/finally with stop() in finally.
  2. Every skill wraps its threads in try/finally with cleanup.
  3. A global threading.Event halts all skills on exit signal.
  4. atexit fires the adapter's authoritative emergency_stop.
  5. GPIO cleanup releases all pins.
