AETHER

The OS layer between LLMs and commodity robots. Talk to any robot in plain English — AETHER finds your hardware, writes the adapters, and runs your objectives safely.

$ pip install aether-robotics

Overview

You type an objective in plain English. The planner picks tools from a registered vocabulary, runs them on your hardware, checks each result, and closes a safety chain when it's done. There's no per-task code to write.

Tools come from three sources: built-in cross-platform primitives, kit adapters (built-in or generated), and parallel skills that coordinate multiple threads.

Features

Plain-English control

Type an objective; the planner turns it into tool calls.

Auto-discovery

Scans I²C, GPIO, serial, camera and network at boot.

Parallel skills

Motors and sensors run at once behind a shared safety event.

Five-layer safety

Every exit path stops motors and releases pins.

Self-correcting

Catches bad tool calls, retries, and verifies results.

Drone + ground

Cars, Mecanum rigs, pan-tilt heads, MAVLink drones.

Lightweight

Runs on a Raspberry Pi 4 — no GPU required.

Universal adapters

Has a Python library? AETHER can drive it.

Three-tier adapters

However custom your robot is, there's a path in.

  1. Known kits — built in. PiCar-X, Freenove, Adeept, PCA9685, L298N, Waveshare. Zero config.
  2. New kits — auto-generated. Claude reads the vendor library, writes an adapter offline, safety-checks it, and registers it.
  3. Custom rigs — guided REPL. A guided REPL maps your hardware onto the standard vocabulary.
›

Generated adapters that reach for tools outside the standard vocabulary fail validation and are rejected.

Tool vocabulary

Every adapter registers a subset of these. The planner only picks from these names.

GroupTools
Motionmove_forward move_backward turn_left turn_right strafe_left strafe_right stop emergency_stop
Sensorsget_distance get_orientation
Pan / tiltlook_up look_down look_left look_right look_center
Genericcapture_image system_metrics get_battery

How it works

  1. You type an objective, in plain English.
  2. The planner picks tools from the registry.
  3. Each call runs on the hardware and returns a result.
  4. The CorrectionAgent checks it — pass advances, failure triggers a fallback chain.
  5. On finish, the safety chain stops motors and releases every pin.

The flow

A single objective, end to end:

You: "drive forward and stop 10 cm before the wall"
  → LLM Planner        picks drive_until_obstacle(target_cm=10)
  → ParallelExecutor   spawns a motor thread + a sensor thread
  → shared should_stop   either thread can fire the stop
  → stop confirmed        CorrectionAgent OK · result returned

Safety chain

A crashed process with live motors is dangerous. Five independent layers each catch a different failure class:

FailureCaught by
Tool exception1 · tool try/finally
Mid-thread exception2 · skill try/finally
Ctrl-C3 · _GLOBAL_STOP
os._exit() / segfault4 · atexit
Pins left claimed5 · GPIO cleanup
