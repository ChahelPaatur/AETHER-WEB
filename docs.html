Features

Everything it does, out of the box.

Eight things AETHER handles so you don't have to.

F.01

Plain-English control.

Type an objective. The planner turns it into tool calls — no per-task code.

F.02

Auto-discovery.

Scans I²C, GPIO, serial, camera and network at boot, then registers the right tools.

F.03

True parallel skills.

One objective, many threads. Motors and sensors run at once behind a shared safety event.

F.04

Five-layer safety.

Ctrl-C, crash or segfault — every path ends with motors stopped and pins freed.

F.05

Self-correcting.

Catches bad tool calls, retries down a fallback chain, verifies the result.

F.06

Drone + ground.

Cars, Mecanum rigs, pan-tilt heads, MAVLink drones — one vocabulary.

F.07

Lightweight.

Runs on a Raspberry Pi 4. No GPU — the Pi just orchestrates.

F.08

Universal adapters.

Has a Python library? AETHER can drive it.

The moat

Three tiers of universality.

However custom your robot is, there's a path in.

1
Known kits — built in

PiCar-X, Freenove, Adeept, PCA9685, L298N, Waveshare. Zero config.

2
New kits — auto-generated

Claude reads the vendor library, writes an adapter offline, safety-checks it, registers it.

3
Custom rigs — guided REPL

A guided REPL maps your hardware onto the standard vocabulary.

⌁

Generated adapters that reach for tools outside the standard vocabulary fail validation and are rejected.

Architecture

How it works.

Five steps, every objective:

1
You type

An objective, in plain English.

2
Planner picks tools

From the registry — built-in primitives, kit adapters, and parallel skills.

3
Hardware runs it

Each call executes on the device and returns a result.

4
CorrectionAgent checks

Pass advances to the next step; failure triggers a fallback chain.

5
Safety chain closes

On finish, motors stop and every pin is released.

Design rationale

Why it's built this way.

Parallel, not serial.

A serial loop jitters — the motor stutters ten times a second. Parallel threads keep the motor smooth while a separate thread watches the sensor. Cleaner motion, accurate stops.

Five layers, five failure classes.

A crashed process with live motors is dangerous. Each layer catches a different way things break:

FailureCaught by
Tool exception1 · tool try/finally
Mid-thread exception2 · skill try/finally
Ctrl-C3 · _GLOBAL_STOP
os._exit() / segfault4 · atexit
Pins left claimed5 · GPIO cleanup
